## 1. Project / Build Name

**Cora AI OS** — multi-agent assistant + AI operating system built on top of self-hosted infrastructure (Ubuntu Server, Docker, Postgres, Redis, n8n, DGX Spark).

Repos:
- `/home/owner/cora-ai-os/` — primary application repo (cora-api, cora-ui, cora-worker, mcp-filesystem, mcp-placeholder)
- `/home/owner/cora-stack/` — pre-existing infrastructure compose (n8n + originally postgres + redis). Postgres has since been moved into cora-ai-os compose with pgvector.

## 2. Primary Goal

Build a governed multi-agent platform where Cora (user persona) is orchestrated internally by ATLAS, with specialist subagents (SCRIBE memory, FORGE engineering, PULSE research, future SIGNAL/CHRONOS). All chat, memory, tools, MCP, plans, jobs, delegations, traces, and workspaces are auditable; admins can govern agents/tools/permissions; non-autonomous by design (deterministic routers + manual triggers + admin actions only).

## 3. Current Build State

**Running services (docker compose in `~/cora-ai-os`):**
- `cora-postgres` — pgvector/pgvector:pg16, container `cora-postgres`, on `cora-internal` aliases `postgres` + `cora-postgres`, external volume `cora-stack_postgres_data`

**Working features:**
- Auth: JWT bearer, register (first user = admin bootstrap), login, /auth/me, all protected routers gated via `Depends(get_current_user)` or `Depends(require_admin)`
- Chat pipeline: memory intent → tool intent (n8n) → FORGE filesystem intent → plan intent → web-search intent (PULSE) → ATLAS routing → current-date/time preamble + workspace context injection → hybrid memory retrieval (semantic + keyword merge) → LLM call → persist + trace + delegations
- Subagents: SCRIBE (memory), FORGE (engineering), PULSE (research/synthesis incl. live `web_search` via SearXNG), **SIGNAL (communication/messaging)**, **CHRONOS (time/schedule/planning)** — DB-backed prompts with Python-constant fallback; deterministic keyword routing in `app/agents/routing.py` (most matches wins, tie → specificity then `_CANDIDATES` order FORGE→PULSE→SIGNAL→CHRONOS, else Cora). Routing keywords prefer the active version's `metadata.routing_keywords` (DB-managed) over the Python constants.
- Current-date awareness: every LLM turn is prefixed with the current date/time (`app/clock.py`, `CORA_TIMEZONE` env, default UTC)
- Memory: scoped (user/global/system) + workspace-aware, legacy NULL fallback, chat-driven intents, SCRIBE summarization, content_hash dedup. **Chunked embeddings:** long content split into overlapping chunks in `memory_entry_chunks` (4000-char/400-overlap), each embedded; `semantic_search` prefers chunk vectors (mapped to parent, injects matched chunk + source citation), falls back to memory-level embeddings then keyword. Memory-level embedding still maintained for compat. Endpoints: `/memory/{id}/embed`, `/memory/embed-missing` (admin), `/memory/{id}/chunks/rebuild`, `/memory/chunks/rebuild-missing` (admin). retrieval_mode labels incl. `semantic_chunks`/`hybrid_chunks`.
- Tool registry with governance: deterministic chat tool router, `/tools` + `/admin/tools`, runtime governance (`allowed_agents`, `risk_level`, policy table, rate limits), audit logs + traces
- **SIGNAL/CHRONOS governed planning (internal-only, review-only):** seeded `internal_action` tools `signal_create_draft` / `chronos_create_schedule_proposal` (requires_confirmation=TRUE) + `communication_drafts`/`schedule_proposals` tables + REST CRUD (`/workspaces/{id}/signal/drafts`, `/workspaces/{id}/chronos/proposals`, owner-scoped, admin sees all) + Admin Console → Agents sub-tabs (Copy Draft/Mark Reviewed/Approve/Archive only — no send/schedule). Status capped at draft·reviewed·approved·archived / proposed·reviewed·approved·archived. Traces: signal_draft_*/chronos_proposal_*. **Chat-to-draft (v0.2):** a routed SIGNAL/CHRONOS chat turn that *explicitly* asks to save creates a draft/proposal via the governance check (`check_permission` + `tool_execution_logs` success/failed/denied + `signal_draft_created_from_chat`/`chronos_proposal_created_from_chat` traces), appends a "✓ Saved as an internal draft/proposal" confirmation, and never sends/schedules; non-explicit turns create nothing; the Agent Test Harness path creates nothing. No generic `internal_action` runner yet (TODO in `tools/registry.py`) — chat calls the service functions directly behind the governance gate. **Review workflow (v0.3):** governed internal lifecycle via `app/review_workflow.py` — statuses draft/proposed→in_review→(changes_requested)→reviewed→approved→archived; per-item endpoints submit-review/request-changes/mark-reviewed/approve/archive/review-events; review columns + `draft_review_events`/`proposal_review_events` audit trail; PATCH is content-only (edit blocked once locked unless admin); **approve requires admin** (no self-approve); 10 `signal_draft_*`/`chronos_proposal_*` lifecycle traces. UI = per-row Manage panel (notes + status-gated actions + history). "Approved" is internal only — still no send/schedule. **Integration Readiness (v0.4):** approved drafts/proposals can spawn a **dry-run-only** `external_integration_intents` record (`app/integration_readiness.py` + `routers/integration.py`) — normalized email/calendar payload preview, validation (hard errors→blocked / warnings→ready_for_confirmation), admin-only internal confirm (dry_run stays TRUE), cancel, full `external_integration_events` trail + 5 `integration_intent_*` traces; seeded readiness tools `signal_prepare_email_send_intent` / `chronos_prepare_calendar_event_intent` (internal_action, medium). UI = Tools → Integration Readiness + "Prepare Send/Calendar Intent" on approved items. **No live providers** — internal_preview only; nothing contacts Gmail/Outlook/Google/Microsoft. **Provider connectors (v0.5):** `external_provider_connectors` registry (6 seeded: internal_preview_* enabled, gmail/outlook_mail/google_calendar/microsoft_calendar disabled — all dry_run_only, supports_send/calendar_*=FALSE) + `app/provider_connectors.py` connector contract (validate/build_preview/execute_dry_run, `LIVE_EXECUTION_ENABLED=False`, zero external-client imports) + `routers/integration_providers.py` (`GET/PATCH /integration/providers`, admin PATCH can't enable live flags) + `POST /integration/intents/{id}/dry-run` (provider payload preview, external_action_performed=false). Intent creation resolves/validates the connector + rejects disabled non-preview providers. UI = Tools → Providers + "Dry Run Provider Payload" in Integration Readiness. Traces: provider_connector_listed/updated, provider_payload_validated, provider_dry_run_executed.
- MCP layer: HTTP JSON-RPC client + REST surfaces on placeholder servers, `/admin/mcp/*` endpoints, filesystem MCP works with the workspace volume mount

## 8. Architecture Decisions Made

**Persona vs orchestrator:**
- `PERSONA_NAME = "Cora"` — user-facing voice.
- `ORCHESTRATOR_NAME = "ATLAS"` — internal routing. Never user-visible except as "Orchestration: ATLAS" metadata.
- Subagents (FORGE, SCRIBE, PULSE) are internal modes. The model is told "you are Cora operating in FORGE mode" (or PULSE mode) when routed.

**Agent registry (DB-backed prompts):**
- `agents` + `agent_versions` tables. Runtime tries DB-loaded active version of the selected subagent's prompt; falls back to Python constants in `app/agents/<agent>.py` if DB fails. Logged as `prompt_source=python_constant | db_agent_version:v<n>`.
- Chat resolves the prompt via the module-level `SUBAGENT_PROMPTS = {FORGE: ..., PULSE: ...}` map in `chat.py` (generalized 2026-05-28 from the old FORGE-only branch). Add future subagents by extending this map + `routing._CANDIDATES` + `registry._SEED_AGENTS`.

**Multi-agent routing (`app/agents/routing.py`, 2026-05-28):**
- `select_subagent(message)` matches each subagent's routing keywords (word-boundary for single words, substring for multi-word phrases), scores by match count, returns the highest scorer; ties broken by `_CANDIDATES` order (FORGE before PULSE); no matches → Cora persona. Purely deterministic, no model call.

**Memory scoping (additive):**
- 3 levels: user/global/system, plus workspace_id (nullable). Legacy rows with NULL scope_id or NULL workspace_id are treated as visible (defensive fallback to avoid breaking pre-scoping data).
- Memory filter: `(scope_type='global' OR (scope_type='user' AND (scope_id=$user OR scope_id IS NULL))) AND (workspace_id=$ws OR workspace_id IS NULL)`.

**Hybrid memory retrieval in chat:**
- Try semantic (if pgvector + embedding model configured), always also run keyword, merge with semantic-first dedupe by id, cap at top 5 for prompt.
- `retrieval_mode` classified: semantic / hybrid / fallback_keyword / keyword / unavailable.

**Workspace context injection:**
- On every LLM-path chat call, compact context (≤1500 chars) injected between system prompt and memory block. Includes workspace name/slug/description + memory counts + active plans + active jobs + enabled agents/tools/MCP servers.

**Tool governance:**
- Precedence: enabled → policy_table (per tool×agent override + rate limit) → tool.allowed_agents → risk_level=high requires admin.
- `agent_name=None` (manual user run) skips allowed_agents check.
- Audit log + runtime trace on every dispatch.

**MCP layer:**
- HTTP JSON-RPC client only in v0.1. Placeholder + filesystem servers respond to both REST (`/health`, `/capabilities`, `/execute`) and JSON-RPC (`/`) so the same client works against either. stdio/SSE explicitly deferred.

**Plans + jobs + worker:**
- Template-based plan generation only (5 generic steps with ATLAS/FORGE assignments). No autonomous execution.
- Worker claims with `FOR UPDATE SKIP LOCKED`, transitions queued → running → completed/failed, retries up to `max_attempts`.
- Worker handler for `execution_plan_step` (legacy alias `plan_step`).
- Step execution opens an ATLAS→assigned_agent delegation if assigned_agent is non-ATLAS.

**Delegations:**
- ATLAS-only orchestrator. Self-delegation rejected. Max 3 concurrent (pending+running) per session OR per plan.
- Auto-recorded in `/chat` (ATLAS→agent on routing + SCRIBE→agent on memory injection) and in worker (ATLAS→agent on step execute).

**Knowledge ingestion:**
- Every entry creates a `knowledge_sources` row first (content_hash dedup, reuse existing source on duplicate with warning), then memory_entries linked via `source_id`.
- Sources support six types: manual_note, markdown, text_file, url, generated_summary, system_seed.

**UI architecture:**
- React 18 + Vite, dark/violet Cora theme, no state library. Sidebar has Chats/Memory tabs + admin menu items.
- Workspace selector in sidebar header persists in localStorage; sent on every chat request.
- Impersonation: admin can issue a token for another user; their own token stashed in `cora_admin_token` localStorage; "Restore admin" banner.

### Worker concurrency — bounded async pool — COMPLETED (2026-06-28)

`cora-worker` processed exactly one job at a time: the main loop called `process_one()`, which claimed a single queued row (`FOR UPDATE SKIP LOCKED`) and `await`ed its handler to completion before looping. After Phase 3 added the worker-driven `agent_run` handler, a single long agent run (DGX latency + spokes) would block the whole queue — news-feed refreshes and any other runs waited behind it, and even the 60s news-scheduler tick only fired between jobs. Backlog item 3 fixed this with a **bounded concurrent pool**, minimally and testably: `process_one` was split into `run_claimed(job)` (the post-claim body — dispatch handler, mark completed/failed, traces, per-job heartbeat; wrapped so a crashed task never raises) plus three extracted, dependency-injected helpers — `_fill_slots(inflight, max_concurrency, claim, run)` (claim+spawn until the pool is full or the queue is empty), `_reap(inflight)` (drop finished tasks, surface a stray exception), and `_idle_wait(inflight)` (sleep until a running job finishes, shutdown fires, or the poll interval elapses — so a freed slot refills immediately, not only on the next poll). `main()` now keeps a set of in-flight tasks, fills it to `WORKER_MAX_CONCURRENCY` (env, default 3) each pass, and runs the heartbeat + news-scheduler tick **alongside** long jobs instead of behind them; shutdown drains in-flight tasks for up to `WORKER_SHUTDOWN_DRAIN_SECONDS` (default 30) before closing the pool. Concurrency is DB-pool-safe: handlers acquire a Postgres connection per query and release it immediately (the long wait is the DGX HTTP call, which holds none), so 3 concurrent jobs stay well under the worker's `max_size=10` pool even with parallel spokes. Default 3 (not 1) deliberately changes behavior — set `WORKER_MAX_CONCURRENCY=1` to restore strict serialization. **Verified:** new `scripts/verify_worker_concurrency.py` (DB-free — drives the pool helpers with in-memory fakes: cap is never exceeded, peak concurrency reaches the cap, every queued job runs once, cap=1 serializes, a crashing task is reaped without wedging the pool, SHUTDOWN halts claiming; 12 assertions) PASS; worker rebuilt + healthy (`max_concurrency=3` in the startup log); `/worker/health` shows the main heartbeat (~2s) and the news scheduler both ticking, the scheduler re-firing on its 60s cadence while the loop cycles.
